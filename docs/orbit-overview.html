Why Orbit exists

Every tool is good. The distance between them is the problem.

A working day already spans a ticket tracker, a terminal, an editor, a mail client, a database console and a container dashboard. Each is fine on its own. None of them share context, so the person in the middle pays for it.

Context lives everywhere

Tickets in one system, tasks in another, sprints in a third — and the whole picture in none of them.

Switching costs compound

Terminal for git, browser for pull requests, editor for code. Each hop is small; the day is not.

Time is reconstructed

Timers sit outside the work they measure, so logged hours get rebuilt from memory at 6pm.

Infrastructure sits apart

Databases and containers live far from the tickets and commits that actually drive them.

Time, focus and team

Your dashboard: greeting, weather, active timer, tasks and insights needing attention. Customise lets you drag tiles to reorder, resize them, and hide what you never look at.

The ecosystem

Orbit is not one program. It is three cooperating layers.

Each layer owns exactly what only it can do — and the split is a security boundary as much as an architectural one. Secrets and machine access stay local; shared state stays hosted.

Orbit on the Web

The hosted interface and backend. Always on, shared across your devices.
netlify/functions

A React 19 and Vite 7 single-page app is the control room itself. Behind it, serverless functions handle the things a browser must never do: OAuth code exchanges, provider proxying with server-side secrets, and scheduled jobs that run whether or not anyone is signed in. Supabase Postgres is the system of record, with row-level security scoping every row to its owner.

The Agent

A small companion process on your own machine. The only part with OS-level reach.
agent/server.mjs

Browsers cannot open VS Code, read your Docker containers or run git. The agent can. It is a single download with no installer — when it is running, the top-bar pill turns green. Most of Orbit keeps working without it; those screens simply show a clear “agent offline” notice instead of breaking.

git status, pull, diff Docker Gmail Postgres project terminal native app launch local AI model

An honest note on what runs where. The Event and Knowledge engines are genuinely live in the browser. The Integration and AI engines are constructed there for a consistent API, but they are deliberately inert client-side: real provider calls need credentials the browser is never given, so they happen server-side or through the agent instead.

Inside the engines src/engines

Four engines, one shared model of your work.

Adding a provider or a model means writing one adapter, not touching call sites. That is the whole point of separating them.

Event Engine

A publish-and-subscribe backbone for immutable domain events. A task moved, a build finished, a ticket raised — each becomes an event, durably stored and then dispatched, so the rest of Orbit can react in order and after the fact rather than polling.

Knowledge Engine

Turns that event stream into an entity and relationship graph: which project a ticket belongs to, who touched it, what it blocks. This graph is what the Intelligence answers and the proactive detectors read from — not a fresh query each time.

Integration Engine

One adapter registry where providers declare typed capabilities instead of being special-cased. Seven capability slots are defined; source control is the one fully built out today, with GitHub, GitLab and Azure DevOps behind a single interface. Status changes publish back as domain events, so a disconnected provider is something the workspace knows about.

AI Engine

A provider-agnostic router that tries your preferred model first and falls back through the rest — Claude Haiku 4.5, Gemini 3.6 Flash, GPT-4o mini and Grok 3 mini. Callers ask for a completion; every provider difference stays inside its adapter. The locally hosted model is reached through the agent rather than this router.

Data and security

Your workspace is yours by default.

Nobody sees your projects, tasks or tickets unless you deliberately share them with a team. That is enforced in the database, not just in the interface.

  • Row-level security Every table scopes rows to their owner in Postgres itself, so a bug in the client cannot widen access.
  • One-time-code auth Sign-up, verification and password reset run on emailed one-time codes — never a reset link — issuing session tokens with the same claim shape row-level security expects.
  • Sign-in alerts A new sign-in emails you the time, location and device it came from, so an unfamiliar one is obvious immediately.
  • Secrets stay server-side Provider tokens and service-role keys live in serverless functions. The browser is never handed a raw provider token.
  • Local credentials stay local Your AI provider keys, Gmail app password and database passwords are held by the agent on your own machine, which listens only to that machine.
  • Everything is recorded Sign-ins, integration changes and work-item edits land in a filterable audit log — including anything an automation rule performed on your behalf.
