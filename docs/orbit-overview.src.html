Why Orbit exists

Every tool is good. The distance between them is the problem.

A working day already spans a ticket tracker, a terminal, an editor, a mail client, a database console and a container dashboard. Each is fine on its own. None of them share context, so the person in the middle pays for it.

Context lives everywhere

Tickets in one system, tasks in another, sprints in a third — and the whole picture in none of them.

Switching costs compound

Terminal for git, browser for pull requests, editor for code. Each hop is small; the day is not.

Time is reconstructed

Timers sit outside the work they measure, so logged hours get rebuilt from memory at 6pm.

Infrastructure sits apart

Databases and containers live far from the tickets and commits that actually drive them.

The ecosystem

Orbit is not one program. It is three cooperating layers.

Each layer owns exactly what only it can do — and the split is a security boundary as much as an architectural one. Secrets and machine access stay local; shared state stays hosted.

Orbit on the Web

The hosted interface and backend. Always on, shared across your devices.
netlify/functions

A React 19 and Vite 7 single-page app is the control room itself. Behind it, serverless functions handle the things a browser must never do: OAuth code exchanges, provider proxying with server-side secrets, and scheduled jobs that run whether or not anyone is signed in. Supabase Postgres is the system of record, with row-level security scoping every row to its owner.

The Agent

A small companion process on your own machine. The only part with OS-level reach.
agent/server.mjs

Browsers cannot open VS Code, read your Docker containers or run git. The agent can. It is a single download with no installer — when it is running, the top-bar pill turns green. Most of Orbit keeps working without it; those screens simply show a clear “agent offline” notice instead of breaking.

git status, pull, diff Docker Gmail Postgres project terminal native app launch local AI model

An honest note on what runs where. The Event and Knowledge engines are genuinely live in the browser. The Integration and AI engines are constructed there for a consistent API, but they are deliberately inert client-side: real provider calls need credentials the browser is never given, so they happen server-side or through the agent instead.

Inside the engines src/engines

Four engines, one shared model of your work.

Adding a provider or a model means writing one adapter, not touching call sites. That is the whole point of separating them.

Event Engine

A publish-and-subscribe backbone for immutable domain events. A task moved, a build finished, a ticket raised — each becomes an event, durably stored and then dispatched, so the rest of Orbit can react in order and after the fact rather than polling.

Knowledge Engine

Turns that event stream into an entity and relationship graph: which project a ticket belongs to, who touched it, what it blocks. This graph is what the Intelligence answers and the proactive detectors read from — not a fresh query each time.

Integration Engine

One adapter registry where providers declare typed capabilities instead of being special-cased. Seven capability slots are defined; source control is the one fully built out today, with GitHub, GitLab and Azure DevOps behind a single interface. Status changes publish back as domain events, so a disconnected provider is something the workspace knows about.

AI Engine

A provider-agnostic router that tries your preferred model first and falls back through the rest — Claude Haiku 4.5, Gemini 3.6 Flash, GPT-4o mini and Grok 3 mini. Callers ask for a completion; every provider difference stays inside its adapter. The locally hosted model is reached through the agent rather than this router.

Built with package.json

A deliberately small stack, chosen per layer.

Three runtimes — a browser app, serverless functions and a local Node process — share one TypeScript codebase. Nothing here is picked for novelty; each choice earns its place by removing a problem the layer above it would otherwise have to solve.

Interface
React 19 Vite 7 TypeScript 5.6 React Router 7

A single-page app, every authenticated route lazy-loaded so the first paint stays small. There is no CSS framework — the control-room look is one hand-written stylesheet of semantic classes, themed entirely through custom properties, which is why theme, accent colour, font and density are all user settings rather than a rebuild.

Data
Supabase Postgres PostgREST Row-level security supabase-js 2

Supabase is used purely as Postgres plus its REST layer — Supabase Auth is not used at all. Access control lives in the database as row-level security, so the browser can talk to the API directly without that being a way around permissions.

Server
Netlify Functions 5 jsonwebtoken bcryptjs nodemailer

Serverless handlers own everything the browser must not: OAuth code exchanges, provider proxying with server-side secrets, and the scheduled brief, digest, anomaly-scan and mail-send jobs. Auth is custom — passwords hashed with bcrypt, sessions signed as JWTs carrying the same claim shape the database policies already expect.

Local agent

The companion process: an Express server on localhost exposing 77 endpoints for git, Docker, terminals, native app launches, IMAP mail, Postgres and AI. It ships as a single executable — bundled with esbuild and injected into the Node runtime with postject, so there is nothing to install.

Vitest covers the engines and pure logic — every engine takes its dependencies by injection, so tests run against in-memory stores rather than a live database. Playwright drives the real UI end to end.

The data model supabase/schema.sql

Twenty-four tables, and one rule that guards all of them.

Everything Orbit knows lives in one Postgres database. The shape is deliberately boring — UUID keys, timestamps on every row, foreign keys that cascade from the owner down — because the interesting part is where the rules live: in the database, not the client.

Where the rules actually live

Four things are pushed into Postgres rather than left to the client, each because the client is the wrong place to be trusted with them.

  • Row-level security Enabled on every table, with policies scoping rows to auth.uid(). Because sessions are signed with the same claim shape Supabase Auth would issue, the policies work unchanged despite auth being custom.
  • Service-role-only writes Team membership and invites have no client-facing write policy at all. They are only reachable through a serverless function using the service-role key, which runs its own authorization checks — that is what stops anyone inserting themselves as owner of an arbitrary team.
  • Triggers that keep data honest A task's completed_at is stamped by the database whenever its status becomes done, so every call site feeds the retrospective and estimate-accuracy views without having to remember to. Separate triggers fan out team-activity and @mention notifications, which a client could not do without write access to other people's rows.
  • Functions instead of round trips merge_user_settings merges a settings patch in one statement, so two concurrent saves cannot clobber each other. orbit_hours totals your tracked time server-side, taking your local midnight as an argument so “today” means your day, not the server's.

Data and security

Your workspace is yours by default.

Nobody sees your projects, tasks or tickets unless you deliberately share them with a team. That is enforced in the database, not just in the interface.

  • Row-level security Every table scopes rows to their owner in Postgres itself, so a bug in the client cannot widen access.
  • One-time-code auth Sign-up, verification and password reset run on emailed one-time codes — never a reset link — issuing session tokens with the same claim shape row-level security expects.
  • Sign-in alerts A new sign-in emails you the time, location and device it came from, so an unfamiliar one is obvious immediately.
  • OAuth tokens stay server-side Repository, Sentry, Teams and cloud tokens live in provider_connections and are used only by serverless functions, which proxy the calls. The browser never holds one.
  • Agent credentials are yours alone The credentials the local agent needs — Gmail app password, database passwords, AI keys — sit in your own row-level-secured rows and are passed to the agent per request; it keeps no copy. They are protected by database access control rather than separate encryption at rest, so treat the database as the sensitive store it is.
  • The agent listens locally It binds to your own machine only, so nothing on the network can drive git, Docker or your filesystem through it.
  • Everything is recorded Sign-ins, integration changes and work-item edits land in a filterable audit log — including anything an automation rule performed on your behalf.
